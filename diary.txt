    원격 repo 사용하기
    
    깃으로 관리되는 프로젝트를 깃에서는 repository 저장소라고 부른다 ( 짧게 해서 repo )
    
    1. 로컬repo :
    
    - 내컴퓨터에 저장되어 있는 깃으로 관리되는 프로젝트를 로컬repo라고 함
    
    1. 원격repo :
    - 깃헙처럼 다른 곳에서 접속할 수 있는 공간에 저장되어 있는 거를 remote repository
    - 저쪽 원격 있는데 저장되어 있다고 해서 원격repo 라고 부름
    - 로컬리포와 원격리포를 연결하는 거를 트래킹이라고 추적한다. 라고 표현을 함 (정확히는 branch tracking)
    - 원격리포를 내컴퓨터에 가져오는것( 원격 리포지토리만 있고 , 내로컬 리포지토리가 없을때 (초기 다운로드 같은 느낌) ) - 클론(복제) , 작업내역까지 같이 가져옴

    ## 정리하기
    
    - 버전관리를 한다는 것은 프로젝트 상태가 변경되는 정보를 알고 있다는 것입니다. Git은 commit을 사용해서 버전이 달라지는 것을 관리합니다.
    - 컴퓨터에 있는 프로젝트를 Git이 관리하는 츠로젝트로 만들 수 있습니다. 앞으로 Git으로 관리할꺼야 ! 하고 설정해주면 됩니다. 이 작업을 git초기화 (git initialize)한다고 표현합니다.
    - 현재 프로젝트의 살태응 찰각 저장하는 것을 commit이라고 합니다.
    - commit 에는 아래를 포함합니다.
        - 누가 (author).언제 commit 했는지의 정보와 프로젝트 변경 내용
        - 작업내역이 어떤 것인지 알아볼 수 있게 적는 메시지를 commit메시지라고 합니다.
    - commit 에 반영할지 안할지는 파일 단위로 선택할 수 있습니다. commit에 반영할 파일을 선택하는 것을 add( 혹은 staging, 스테이징) 이라고 합니다.
    - commit한 기록은 history로 볼수 있습니다.
    - 지금까지 우리가 한작업은 git초기화하기(initialize) - add(staging) - commit 입니다.
        - git 초기화는 처음에 단 한번만 해주면 된다. 작업 내역을 저장하기 위햐서는 add - commit만 하면 된다.

- 2주차
    
    issue 가 뭔가요? - 개념 탑재  🔥 위에서 정한 작업방식 중 1단계를 issue를 사용해 작업할 수 있습니다. 👉 1단계. 누가 이 작업 할 것인지 정한다. - Issue 2단계. 각자 맡은 것을 작업한다. - Branch 3단계. 각자 작업을 프로젝트에 합친다. - merge (경우에 따라). 작업한 내용을 리뷰하고 최종적으로 프로젝트에 반영한다. - PR 후 merge
    
    나의 작업공간  브렌치라고 하고 각자 작업을 프로젝트 합치는것을 머지라고한다.
    
    Pull Request : 제안사항
    
    브랜치 이름 : 규칙을 정해서 작성
    
    보통 기능같은걸 만들때는 feature/2(이슈번호)_(잘나타낼수있는이름)  Ex) feature/2_jjigae
    
    브렌치를 하게되면 해당 커밋에서부터 내가 갈라져 나왔다는 이야기


- 협업할떄의 단꼐
	1단계 : 누가 이작업을 할 것인지를 정한다 - Issue
	2단계 : 각자 맡은것을 작업한다 - Branch
	3단계 : 각자 작업을 프로젝트에 합친다 - Merge
	(경우에 따라) 작업한 내용을 리뷰하고 최종적으로 프로젝트에 반영한다 PR후 Merge
- Merge
	브렌치의 어떤 작업내역 커밋들을 다른 브렌치로 합치는 것을 머지 혹은 병합이라고 한다.
	기본적으로 브렌치 단위로 머지하게 됨
	기준이 되는 메인브렌치를 정해놓고 다른 브렌치들을 메인 브렌치에 머지하는것이다.
	!! 머지 충돌해결중에 갑자기 없었던 새로운내용을 넣으면 안됨 !!!! 새로운 내용을 넣고자 한다면 먼저 기존의 내용들을로 충돌 해결한후에 새로운 커밋으로 해결한다 !!!

새로운 브랜치에 커밋을 한뒤 푸쉬를 해야 깃에 오픈 리퀘스트가 가능함
Pr하는 단위는 브랜치 단위 
Pr이 다끝나면 브렌치를 삭제한다
다른사람의 오픈소스에 포크해서 내가 가져와 풀 리퀘스트를 통해 내가 추가한 코드를 머지 요청을 할수가 있다 (이슈번호를 필이 기입해준다)
Pr하기전에 꼭 먼저 코멘트를 남겨준다
Pr전에 꼭 먼저 코멘트를 남겨준다
Pr에는 이슈번호를 포함시켜준다
Pr이 완료되기전 까지 Pr요청한 로컬 브랜치를 삭제하지 않는다

Amend (고치기) 로는 가장 최신의 코밋만 고칠수 있다
커밋을 되돌리기 Revert
이 커밋으로 초기화 Mixed : 작업상태는 그대로 두지만 인덱스는 리셋
		Hard : 모든 작업상태 내 변경사항을 버림 (위험)
		Stash : 임시 저장했다가 다시 꺼내서 쓸수 있다.
!! 커밋 되돌리는 작업은 나 혼자만 사용하는 브랜치에서만 한다 !!
